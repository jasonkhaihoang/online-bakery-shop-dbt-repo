# Phase 2 — Run Business Intent

> Source: https://www.notion.so/8c518ff8c53a49458088d2502926c0b4

_Business-driven. Run once per business intent (e.g. sales report, sales funnel, customer cohort)._
_Input: business intent + **`project_standards.md`**. Output: dbt models + tests + artefacts. **Tests are generated** (unit, integration, E2E), then **executed together** via `dbt test` to ensure logical correctness._

## 2. Business Logic Testing

_Section 2 generates comprehensive tests (unit, integration, E2E). All tests (2.1, 2.2, 2.3) are mandatory. Tests are generated by each skill, then executed together via single `dbt test` command to ensure models are logically correct before completion._

| Skill | Name | Description | Decided by the skill | Input | Generated artefacts |
|-------|------|-------------|----------------------|-------|---------------------|
| 2.1 | **generating-dbt-unit-tests** — _Generate unit tests for business rules and column constraints._ | Use to generate unit tests for model business rules and column constraints (based on requirements from entry point). **Always generate** as part of entry point orchestration; tests are executed by `dbt test` command. | Test package selection (dbt_utils, dbt_expectations, etc.), fixture strategy, unit test cases covering business rules and edge cases, test coverage requirements | Confirm or extend proposed unit tests, suggest additional edge cases or fixtures needed. | • **Docs:** `/docs/testing/unit_tests.md`<br>• **Config:** `packages.yml`<br>• **YAML:** `models/**/schema.yml` (generic tests)<br>• **SQL:** `seeds/**.csv` (fixtures, when applicable)<br>• **Macros:** `macros/tests/**.sql` (custom assertions) |
| 2.2 | **generating-dbt-integration-tests** — _Generate cross-model relationship validation tests._ | Use to generate integration tests for relationships and data consistency across multiple models within the same intent. **Always generate** as part of entry point orchestration; tests are executed by `dbt test` command. | Cross-model relationships and cardinality rules, reconciliation logic, severity assignment per invariant, integration test cases covering relationships and multi-model edge cases | Confirm or extend proposed integration tests, suggest additional cross-model invariants to validate. | • **Docs:** `/docs/testing/integration_tests.md`<br>• **Config:** `selectors.yml`<br>• **SQL:** `tests/**.sql` (reconciliation, cardinality, multi-model invariants)<br>• **Macros:** `macros/**` (reusable reconciliation/join helpers) |
| 2.3 | **generating-dbt-e2e-tests** — _Generate end-to-end pipeline validation tests._ | Use to generate E2E tests validating the full pipeline from raw source through to final mart output against known reference values. **Always generate** as part of entry point orchestration; tests are executed by `dbt test` command. | Critical business journeys to validate (e.g., raw → staging → intermediate → mart), reference datasets, tolerance thresholds, validation environment and frequency | Confirm or extend proposed E2E validation journeys and reference data. | • **Docs:** `/docs/testing/e2e_validation.md`<br>• **Config:** `selectors.yml`<br>• **SQL:** `models/validation/**.sql`<br>• **SQL/YAML:** `tests/**.sql` and `models/validation/**/schema.yml` |

## Rollout Checklist (run per business intent. All mandatory items below execute automatically)

**Mandatory (always generate):**
- [ ] **1.1** — Design dimensional schema, generate Gold/Mart models, and define business metrics
- [ ] **1.2** — Generate staging and intermediate models with data contracts
- [ ] **1.4** — Generate data quality checks and validation rules (generated, not executed in Phase 2)
- [ ] **2.1** — Generate unit tests for business logic
- [ ] **2.2** — Generate integration tests for cross-model validation
- [ ] **2.3** — Generate E2E tests for full pipeline validation
- [ ] **`dbt test`** — Execute all generated tests (2.1 unit + 2.2 integration + 2.3 E2E) and verify logical correctness

**Optional:**
- [ ] **1.3** — Generate dbt Semantic Layer and metrics store (only if using Semantic Layer)

**Post-Intent Refactoring:**
- [ ] **3.1** — Run after 3+ business intents to consolidate patterns (Phase 3, separate run)

> **Tip:** Requires `project_standards.md` as context (always loaded). All skills read this file before generating any artefact. **Tests (2.1, 2.2, 2.3) are generated, then executed together via single `dbt test` command.** DQ checks (1.4) are generated for future production use but not executed during Phase 2 development. When a new business intent arrives, invoke the master skill `processing-business-intent-to-dbt-artefacts` to run the full sequence.
